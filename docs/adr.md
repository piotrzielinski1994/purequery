# Architectural Decisions — purequery

Append-only log of architectural and design decisions made during development.

## Format

Each entry follows this structure:

| Date | Decision | Rationale |
|------|----------|-----------|
| {YYYY-MM-DD} | {What was decided} | {Why this choice was made} |

| 2026-06-21 | `ConnectionConfig` becomes an engine-tagged discriminated union (network variant host/port/database/user/password for postgres/mysql; sqlite variant `{ file }`); Rust side `#[serde(tag="engine")]` enum; one `connectionOf(node)` helper centralises extraction | First engine that does NOT fit the flat host/port/user/password shape. | 2026-06-21 | REVERSE "Connect is stateless": hold a pool per database id in a process-wide `static LazyLock<Mutex<HashMap<id, HeldConnection>>>` (pool + engine), opened on connect, closed on disconnect; commands address a connection by id, `config` sent only on connect | The 2026-06-20 stateless ADR explicitly said "revisit when SQL execution lands" - it has (transactions + cancel need a held connection). Chose a `static` registry over a Tauri-managed `State<>` to mirror requi's cancel-registry shape and keep `db.rs` command fns as free functions (no `State` param threaded through ~8 commands). `AnyPool` is a cheap `Arc` clone + `DbEngine` is `Copy`, so `with_pool(id)` clones the handle out of the lock and never holds the mutex across an `.await`. Engine travels in the registry because commands no longer carry `config`. Churn: every command gained a `connection_id` first arg, ~10 FE test suites re-pointed from `config` to the id |
| 2026-06-21 | Multi-statement runs are user-owned transactions, NOT auto-wrapped; `execute_sql` returns `Vec<QueryOutcome>` (one per statement) run in order on ONE acquired connection | A DB client must let users write their own `BEGIN`/`COMMIT`/`ROLLBACK`/`SET`; auto-wrapping every batch in a transaction would change semantics (DDL, VACUUM) and surprise them. The only requirement is same-connection execution (`pool.acquire()` once, loop), which a held pool enables. A failing statement stops the batch; prior statements stay applied unless the user wrapped them. Per-engine classify/wrap/execute moved from `&AnyPool` to `&mut AnyConnection` so the whole batch shares the connection |
| 2026-06-21 | Cancel aborts at the next await point via `tokio::select!` against a per-request `CancellationToken` (ported from requi), NOT a server-side `pg_cancel`/`KILL QUERY` | Matches requi's proven pattern (registry + `CancelGuard` Drop + `CANCEL_SENTINEL`) and is enough for the UI affordance (Run becomes Cancel). A true server-side kill needs a SECOND connection issuing `pg_cancel_backend`/`KILL QUERY` - deferred as YAGNI for v1; a statement already sent to the DB keeps running until it yields. Documented limitation, not a bug |
| 2026-06-26 | Postgres-only schema level in the sidebar (MySQL "schema"==database, SQLite none stay flat), addressed end-to-end as `(schema, table)` - NOT display-only grouping | A Postgres DB has many schemas; the flat catalog collapsed `public.users` and `analytics.users` into one ambiguous leaf and every table command hit whichever the server `search_path` resolved first - a silent wrong-table bug. Full qualified addressing (catalog returns `TableRef{schema,name}`; `fetch_table`/`count_table`/`apply_mutations` take `schema: Option<String>`; PG introspection pinned to `table_schema = $schema`, not search_path) makes same-named tables independent. Display-only grouping was rejected: it leaves the addressing bug intact. Uniform schema level + cross-DB browse rejected as artificial / bigger scope |
| 2026-06-26 | Schema is RENDER-DERIVED on `TableNode.schema: string\|null` (not a persisted `SchemaNode`), AND rendered FLAT as a leaf-label qualifier - NOT a nested expandable schema row | Two-part. Backend qualified addressing unchanged - sidebar-rendering only. Same change fixed SQL autocomplete to a nested `schema->table->columns` namespace + `defaultSchema=public` (the prior name-keyed flat map let `vehicle_listing.users` clobber `public.users`, suggesting an unqualified table that failed to resolve) |
| 2026-06-26 | Multi-theme = appearance MODE (light/dark/system) + per-mode SPARSE color overrides, persisted SPLIT across `settings.json` (mode) and a separate `theme.json` (colors, key "colors") - mirroring sibling `requi` | A DB client wants OS-following appearance + the ability to tweak individual tokens. The settings.json/theme.json SPLIT (colors stripped from the settings.json payload on save, recombined on load) makes a color scheme device-syncable on its own, matching requi's keymap-split shape. | 2026-06-27 | MongoDB (first DOCUMENT engine) gets a SEPARATE `src-tauri/src/mongo.rs` module + the official `mongodb` crate, NOT a `db.rs`/`sqlx::Any` arm; `lib.rs` is a dispatcher that routes each connection-addressed command to the Mongo path when `mongo::is_connected(id)` and to the SQL path otherwise | `sqlx::Any` has no Mongo support and Mongo is not SQL, so a fourth `DbEngine` arm was impossible. | 2026-06-28 | MongoDB Extended JSON type wrappers (`$oid`/`$numberLong`/`$numberInt`/`$numberDouble`/`$date`) decoded in filters + pipelines so an ObjectId `_id` is queryable as `{"_id":{"$oid":"..."}}` | A plain-JSON filter could never match an ObjectId `_id` (string != ObjectId) - the seeded `users` collection surfaced this immediately. A single-key object whose key is a known `$`-wrapper decodes to the BSON type; query operators (`$gt`) and multi-key objects stay plain documents. A malformed wrapper (bad `$oid` hex) is a clear filter error, not a silent literal. Rejected auto-coercing any 24-hex string (magic, can corrupt a legit hex string `_id`) |
| 2026-06-27 | The Mongo path REUSES the shared IPC structs (`TableRef`/`TableRows`/`TableColumn`/`QueryOutcome`/`RowMutation`) and the ONE `DataGrid`; documents are flattened DBeaver-style (top-level keys -> columns, `_id` first + primary key, nested object/array -> compact JSON text in-cell, scalar -> JSON-literal text, missing key -> NULL cell) | Honors the non-negotiable ONE-DataGrid invariant (CLAUDE.md) so the frontend grid/paging/sort/count/Save pipeline stays engine-agnostic - the table card renders a collection exactly like a SQL table. Forking a second document grid was rejected. A new shared `RowMutation::Replace { pk_value, document }` variant carries the full-document edit; the SQL `build_mutation` rejects it (`Err`), the Mongo path maps cell/insert/delete/replace to updateOne-$set/insertOne/deleteOne/replaceOne. Cell + insert values are parsed as JSON literals (number/bool/null/string) so BSON scalar types survive an edit; an `_id` that parses as an ObjectId matches as one, else by its raw value. The crate-wide serde_json `arbitrary_precision` feature forced building BSON via a hand-rolled `Value -> Bson` map instead of `bson::to_document` (which emits a `$serde_json::private::Number` wrapper Mongo rejects). Per-engine UI differences (History preview strings, filter syntax) live behind one `queryPreview(engine)` strategy + a `mongodb` JSON CodeMirror mode, not `engine===` branches in the table card. Mongo card shows Query (find/aggregate JSON) + Settings only; no SQL/Views/Script. New Rust deps: `mongodb`, `futures-util` |
| 2026-07-09 | F13 FK navigation gets an explicit back/forward history (a `(tableId, filter)` stack + cursor in `WorkspaceProvider`, pure reducer in `nav-history.ts`, Back/Forward buttons + `Mod+[`/`Mod+]`), REVERSING the plan's original "no stack - rely on the tab bar as the visit trail" | The tab-bar-as-trail choice was made for minimal diff, but the user rejected it after using the feature: after a chain of FK jumps the tab bar shows every target as a peer tab with no notion of "where I came from", so returning is a hunt, not a Back. A browser-style linear history (push on jump, drop-forward-on-new-jump, restore tab+filter on back/forward) matches the mental model of drilling through relations. Kept in-memory only (not persisted) - navigation context is ephemeral like `tableFilters`. The pure reducer isolates the stack semantics from the provider so it is unit-tested without a webview |
| 2026-07-09 | The Postgres FK introspection query reads the referenced side from `key_column_usage` correlated by `position_in_unique_constraint`, NOT from `information_schema.constraint_column_usage` | The original F13 query joined `constraint_column_usage` on constraint_name alone, which for a COMPOSITE FK is a cartesian product (a 2-column FK yields 4 rows: every local column paired with every referenced column), so the folded FK carried duplicated columns and the "Go to" label read `region=EU, region=EU, code=W1, code=W1`. Correlating by position yields exactly one referenced column per local column in order. A single-column FK never exposed this (1x1=1). The unit test that "verified" fold correctness used hand-crafted correct tuples, so it missed the query bug - the fix pins the query text itself (must not contain `constraint_column_usage`, must correlate by `position_in_unique_constraint`) |
| 2026-07-10 | F18 Session Logs tab feeds off tauri-plugin-log's `TargetKind::Webview` (a `log://log` event the FE reads via `attachLogger`), NOT a per-dispatcher `app.emit` at each `lib.rs` log call site | The alternative - emitting a second event next to each existing `log::info!/error!` - would duplicate the emit at ~6 sites, fight the standing "one call site per command" logging rule, and be silently forgotten on every new command. The Webview target is a single additive line in `logging::init` that forwards EVERY record (backend + the FE `log_message` bridge) with zero call-site churn, and the file target is untouched (each line still written once). Cost: one JS dep (`@tauri-apps/plugin-log`) + accepting the plugin's own line formatting. A "read the file on open + Refresh" design was rejected (no live tail; needs a new read-back command); live-only push is the chosen trade-off, with the documented gap that pre-mount startup lines (`purequery starting`, launch auto-connect) are not shown |
| 2026-07-10 | F12 commit/rollback confirmation is a FRONTEND confirm-before-send gate (per-database `confirmWrites` flag, dialog shows the exact SQL, Commit sends / Cancel discards), NOT a real backend manual-commit transaction | The todos.md F12 title is "confirmation layer" and its dominant phrasing is "write NIE leci od razu - trafia do warstwy potwierdzenia z widocznym SQL i przyciskami Commit/Rollback" (held-then-confirmed, not executed) and "Pairuje z F11" - F11 (read-only) is a no-backend-change frontend boundary gate. A true DBeaver manual-commit (BEGIN on a pinned connection, hold across the session, COMMIT/ROLLBACK; Mongo sessions/transactions) needs backend transaction-state on the held pool (only 5 connections, one pinned per manual-commit DB) and has NO requi/vidui precedent to mirror. Chose the frontend gate: mirrors F11 byte-for-byte (persisted `confirmWrites` on the node, `mergeConfirmWrites`, dialog gate on the same two write paths - table Save + write-shaped SQL Run), zero backend risk, safe to complete unattended. `readOnly` takes precedence (a read-only DB blocks outright, never shows the Commit dialog). The real backend manual-commit remains a separate, larger future feature |
| 2026-07-11 | REVERSE the 2026-07-10 F12 confirm-gate decision: F12 becomes REAL manual-commit transaction mode (auto-commit OFF), not a frontend confirm-before-send dialog | The user clarified against a DataGrip/DBeaver screenshot that the ticket wants real manual-commit - writes hit the DB immediately but inside an open transaction, a SELECT sees the uncommitted rows, and a Commit/Rollback toolbar finishes the tx at the DB level. The confirm-gate (hold the SQL, send on Commit, nothing sent on Cancel) was the wrong interpretation. Real design: a per-connection transaction-session registry (Unit of Work) pins a `PoolConnection` from the held pool and issues literal `BEGIN` on the first write; while open, EVERY connection-addressed SQL command (reads + writes) routes through that pinned connection (so uncommitted rows are visible) instead of a fresh pool connection - the bulk of the change is refactoring db.rs read/mutation helpers from `&AnyPool` to `&mut AnyConnection`. `commit_transaction`/`rollback_transaction`/`begin_transaction`/`transaction_state` are new commands; disconnect force-rolls-back to avoid leaking the pinned connection. SQL-only (Postgres/MySQL/SQLite) - MongoDB excluded, mirroring DBeaver's NoSQL driver having no manual-commit toolbar (Mongo tx needs a replica set the standalone test-stack lacks). `BEGIN`-on-first-write (not on entering the mode) keeps pure browsing on the pool so no idle-in-transaction connection is held. This supersedes the frontend `confirmWrites` flag, which is replaced by `manualCommit` on the node |
| 2026-07-12 | Manual-commit (F12) wraps each statement/mutation in a per-statement `SAVEPOINT` inside the open transaction, so a failed statement does not poison the whole tx | On Postgres a failed statement marks the entire transaction aborted ("current transaction is aborted, commands ignored until end of transaction block") - every later command fails until ROLLBACK, so a user who hit one bad write was stuck. DBeaver avoids this with "use savepoint for each statement". purequery now does the same: inside the pinned tx connection `run_one_in_savepoint`/`execute_mutation_in_savepoint` issue `SAVEPOINT purequery_stmt` before each statement, `RELEASE` on success, `ROLLBACK TO SAVEPOINT` on failure (undoing only that statement, keeping the tx usable). Gated on `ConnHandle::is_pinned` - the pooled/auto-commit path takes no savepoint (a savepoint outside a transaction is an error). The batch still stops at the first error (unchanged contract), but the NEXT Run works. Verified red-green on real Postgres (`live_pg_savepoint_keeps_tx_usable_after_a_failed_statement`): without the savepoint the recovery insert returns the aborted-tx error; with it, it succeeds and commits. SQLite does not abort a tx on statement error, so its in-process test is only a savepoint-SQL smoke. |
| 2026-07-16 | F16 backup (export): the backend spawns the engine's system dump tool via `tokio::process` directly (pg_dump/mysqldump/mongodump on PATH), NOT bundled sidecars; restore is deferred to a separate feature | Bundling the dump tools (vidui's ffmpeg-sidecar pattern) was rejected: pg_dump's archive format is tied to the server major version, so a bundled binary would silently produce incompatible or refused dumps against a differing server - a version-matching maze plus large per-engine binaries. Requiring the tool on PATH matches DBeaver/pgAdmin and keeps the app small; a missing tool is a clear runtime `<tool> not found on PATH` message. Spawning from our own Rust command needs no plugin-shell/capability (Tauri capabilities gate JS->Rust invokes, not Rust std::process). Credentials never ride the argv (PGPASSWORD/MYSQL_PWD env, Mongo percent-encoded --uri) so a password can't leak to the process list. Backup takes a fresh config like connect_database (the pool holds none) and needs no open connection - the dump tool connects itself. Progress reuses the existing backend log -> Logs-tab stream (no new channel; no in-app cancel/progress bar in v1). Restore (destructive drop/overwrite) was split out to bound this branch to the safe, additive half. Save-path picking added `tauri-plugin-dialog` (mirrors requi/vidui). SQLite is a plain file copy (no tool) |
