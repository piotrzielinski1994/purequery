# Learnings

Project-specific conventions, gotchas, and constraints worth recording so future-you (human or agent) doesn't re-derive them. Append-only. For architectural trade-offs use [adr.md](adr.md) instead.

- TanStack Hotkeys split: `@tanstack/hotkeys` is the framework-agnostic core (no React hook). The React `useHotkey` + `HotkeysProvider` live in `@tanstack/react-hotkeys`. Install the adapter, not the core.
- Hotkey strings are case-sensitive in the typed union: use uppercase keys, e.g. `"Mod+K"` not `"Mod+k"`.
- Under jsdom the hotkeys lib resolves `Mod` to `Control` (test platform reports non-mac), so hotkey tests fire `{Control>}k{/Control}`, not Meta.
- ESLint react-hooks v7 false-positives: `useReactTable` trips `react-hooks/incompatible-library`; code-based TanStack route files trip `react-refresh/only-export-components`. Both scoped off in eslint.config.js for the relevant paths.
- shadcn Button keeps `react-refresh/only-export-components` as an accepted warning (canonical upstream file exports `buttonVariants` alongside the component). Lint exits 0 with warnings.
- Bootstrap scaffold was ported from the sibling `requi` repo (same Tauri 2 + React 19 + TanStack stack), not generated fresh from `npm create tauri-app`. Identity rewritten: package name, Cargo `name`/`_lib`, `tauri.conf.json` productName/identifier, `main.rs` lib ref.
- shadcn `resizable` ships for the old `react-resizable-panels` API: it passes `direction` to `ResizablePanelGroup`, but v4 renamed the prop to `orientation` ("horizontal"|"vertical"). Fix the prop at call sites or typecheck fails.
- react-resizable-panels v4 `Panel` size props (`defaultSize`/`minSize`/`maxSize`) read a bare `number` as PIXELS, not percent. `defaultSize={20}` = 20px wide. For proportional panels pass a string with a unit: `defaultSize="20%"`.
- jsdom has no `ResizeObserver`; radix Select/Tabs + react-resizable-panels need it. A no-op `ResizeObserver` stub is installed in `src/test/setup.ts`. Also expect harmless "Not implemented: Window's scrollTo()" noise from shadcn `ScrollArea` under jsdom.
- radix `SelectValue` renders nothing until the dropdown opens (items live in an unmounted Portal), and jsdom can't open it. To assert the current value in tests, render the value as explicit `SelectTrigger` children instead of relying on `<SelectValue/>`.
- A tree query leaf's `aria-selected` tracks `activeQueryId` (the active tab), not `selectedNodeId` - clicking a folder must not visually deselect the active query (spec E-2). Folder rows track `selectedNodeId`.
- `getByText` regexes for a panel's empty-state must not also match a nearby `Select` trigger label: the "none" connection body said "No connection auth" while the type select also offered "No Auth", so the test matched two nodes. Renamed the select option to "None" to disambiguate.
- A `react-resizable-panels` group mounted anywhere in a `radix Tabs` subtree breaks tab-switching under jsdom: `user.click`/`fireEvent.click` on a `TabsTrigger` becomes a no-op (state never commits), regardless of `forceMount` or whether the group is inside `TabsContent`. Its global pointer/resize handlers interfere with the click. Consequence: keep resizable splits at the shell level (sidebar|content, content|console) and use a plain fixed flex split inside a tab panel (e.g. the SQL tab's editor|results). Don't nest a ResizablePanelGroup under tabbed content.
- Dual-action tree row (chevron toggles children, row body opens a tab): make the chevron a real `<button aria-label="Toggle <name> tables">` with `onClick` calling `event.stopPropagation()` before toggling, nested inside the clickable `role="treeitem"` div. Without stopPropagation the row's open-handler also fires. Tests target the two actions separately: `within(row).getByRole("button", {name:/toggle .* tables/i})` for expand, clicking the treeitem for open.
- This is a DB client ported from the HTTP-client `requi`: watch for HTTP-isms that don't fit. The auth model came over as a `none/bearer/basic` union -> mapped to `none/password/token` and surfaced as an "Auth type" select. Wrong for databases: a DB connection is host/port/database/user/password (fixed fields), not a tagged auth union. Dropped the `Connection` union entirely; `DatabaseNode` now carries those fields flat. Lesson: when reusing requi's structure, re-judge each domain concept, don't 1:1 map HTTP semantics.
- Dev server port is 1431 (not the create-tauri-app default 1420), pinned in BOTH `vite.config.ts` (`server.port`) and `src-tauri/tauri.conf.json` (`build.devUrl`) - they must match or `tauri dev` hangs "Waiting for your frontend dev server". Tauri does not interpolate env vars in `devUrl`, and `tauri dev --port` controls a different (built-in) server, so a single env override can't drive both without a wrapper (see tauri issue #6186); a hardcoded matching pair is the simplest fix.
- `cmdk` calls `Element.scrollIntoView` when navigating its list; jsdom has no such method, so a `command-palette` test using cmdk throws without a stub. Added a no-op `Element.prototype.scrollIntoView` in `src/test/setup.ts` (next to the `ResizeObserver` stub).
- Command palette is `cmdk` + a radix `Dialog`, reusing the **`radix-ui` umbrella** (`import { Dialog as DialogPrimitive } from "radix-ui"`) like `ui/tabs.tsx`, NOT the standalone `@radix-ui/react-dialog` that `requi` uses. Only `cmdk` was added as a new dep. The radix `Dialog` needs a `DialogTitle` + `DialogDescription` rendered INSIDE `DialogContent` (sr-only) for the accessible name, else radix logs missing-title warnings; `ui/command.tsx`'s `CommandDialog` does this.
- cmdk filters by hiding: a non-matching `CommandItem` returns `null` (unmounts), so tests can assert `queryByText(name).not.toBeInTheDocument()` after typing. `CommandEmpty` renders only when the filtered count is 0.
- `sonner`'s `<Toaster theme="system">` calls `window.matchMedia` on mount; jsdom doesn't define it, so any test rendering `WorkspaceLayout` (which now mounts `<Toaster/>`) throws "window.matchMedia is not a function". Stubbed unconditionally in `src/test/setup.ts` (an `"in" guard does NOT work - jsdom has the property as a throwing getter, not absent). Mock `sonner` with `vi.mock("sonner", () => ({ toast: { success: vi.fn(), error: vi.fn() } }))` to assert toast calls.
- A status indicator with an `aria-label` rendered INSIDE a `role="treeitem"` leaks into the row's computed accessible name (name-from-content), so `getByRole("treeitem",{name:"admin_db"})` breaks once a "admin_db connected" dot appears. Pin the row name with an explicit `aria-label={node.name}` on the treeitem (TableRow already did this); the dot stays independently queryable via `getByLabelText(/admin_db connected/)`.
- Radix `Select` can't be opened under jsdom (Portal + no layout), so a test can't click an option to assert engine-switch wiring end-to-end. Cover the value-follows-state leg by seeding/asserting the trigger text; trust `onValueChange` by inspection (or move to an e2e/browser test). The connect form seeds the engine from `DatabaseNode.engine` and renders the label as explicit trigger children (not `<SelectValue/>`).
- Tauri command args: `invoke("connect_database", { config })` - the object key (`config`) must match the Rust command's parameter name exactly, else deserialization fails at runtime (typecheck won't catch it). Serde enum `#[serde(rename_all="lowercase")]` maps Rust `Postgres`/`Mysql` <-> TS `"postgres"`/`"mysql"`.
- `sqlx::Any` driver can't decode every native column type. Postgres `information_schema.table_name` is the `name` type -> `fetch_all` via `AnyPool` fails at runtime (not at connect) with "error in Any driver mapping: Any driver does not support the Postgres type PgTypeInfo(Name)". Fix: cast to a portable type in the query (`SELECT table_name::text ...`). The connection itself succeeds; the error surfaces only when decoding the result row. Caught in manual testing against a real PG (DBeaver connected fine; our query errored). Pure unit test pins the `::text` cast since no live DB in CI.
- The `Any`-decode trap generalizes to ALL table content: arbitrary user columns (uuid, timestamptz, numeric, jsonb, ...) are equally undecodable. Strategy for browsing rows: read column names from `information_schema.columns` (parameterized, ordinal order), then `SELECT <each col>::text` (PG) / `CAST(<col> AS CHAR)` (MySQL) so every value comes back as a portable string. Decode as `Option<String>` to preserve NULL. Quote identifiers per engine (`"x"` PG / `` `x` `` MySQL, doubling the quote char) to stay injection-safe since table/column names can't be bind params. Pure query-builder fns (`rows_query`/`columns_query`/`quote_identifier`) are unit-tested; the DB round-trip isn't (no live DB in CI).
- Adding `useQuery` to a workspace component means its tests need a `QueryClientProvider` wrapper (with `retry:false` so rejected fetches surface immediately instead of retrying). Mock `@/lib/tauri`'s fetch fn; `mockReturnValueOnce(new Promise(() => {}))` (never-resolving) asserts the loading state.
- `useReactTable` requires STABLE `data` and `columns` references. Building them inline (`rows.map(...)`, `columns.map(columnHelper.accessor...)`) returns fresh arrays each render; combined with any state-driven re-render of the grid this triggers TanStack Table's infinite re-render loop, which pegs the main thread and FREEZES the whole app (clicks/keyboard dead) - it manifests as "nothing is clickable", not a console error. It stayed dormant while the grid had no state; adding row-selection (`useState`) was the first thing to re-render it and exposed it. Fix: wrap both `data` and the column defs in `useMemo`, and memoize the arrays passed in by parent wrappers too (a parent that `.map()`s fresh arrays every render defeats the child's memo). The tiny test fixtures (2-3 rows) did NOT reproduce the loop, so unit tests passed while the real app froze - a reminder that jsdom won't catch a perf/loop bug.
- The table card and SQL result pane share ONE grid (`data-grid.tsx` `DataGrid`) - they must always look identical (user-enforced invariant, see CLAUDE.md). It's edit-capable; read-only callers pass `editable={false}` + no-op handlers. Headers always render (even with zero rows) so an empty result still shows its column structure with "No rows." beneath. Don't fork a second grid.
- A resizable split INSIDE a radix Tabs panel can't use `react-resizable-panels` (its global pointer/resize handlers break tab-switching under jsdom - see the older entry). The SQL editor|results split is a hand-rolled `HorizontalSplit` (`horizontal-split.tsx`): percentage width clamped 15-85%, `window` pointermove/pointerup drag, `role="separator"`. jsdom returns `getBoundingClientRect().width === 0`, so the drag math no-ops in tests unless you mock the rect - the split tests stub `getBoundingClientRect` to assert clamp/drag/release.
- The `useReactTable` stable-ref freeze recurs in EVERY grid, not just the table browser. It bit `OutcomeGrid` (SQL-tab results) too: after Run, `addHistoryEntry` mutates the workspace context value -> SqlEditor re-renders -> `OutcomeGrid` rebuilt `data`/`columns` as fresh arrays -> infinite re-render -> whole app frozen (query had already succeeded, 38 rows visible, then dead). Any NEW grid must `useMemo` its `data` AND column defs from the moment it's written - don't wait for a state-driven re-render to expose it. Rule of thumb: if a component calls `useReactTable`, its `data`/`columns` args must be memoized, period. with `role="separator"`+`aria-label` leaks into the columnheader's accessible name (same name-from-content trap as the sidebar status dot), breaking `getByRole("columnheader",{name:"id"})`. Make the handle `aria-hidden` (it's a mouse affordance, not a named control) and target it in tests via `data-testid`.
- Server-side row filter must bind the pattern as a param, never interpolate: `<col>::text ILIKE $1` (PG, case-insensitive) / `CAST(<col> AS CHAR) LIKE ?` (MySQL). PG reuses `$1` for every OR'd column (one bind); MySQL needs one `?`+bind per column. Validate the filter column against the known column list and fall back to all-columns if unknown - the column name can't be a bind param. Builder returns `(sql, Vec<bind>)` so it stays pure/unit-testable.
- Running arbitrary user SQL through `sqlx::Any`: you don't know the result columns up front and Any can't decode native types, so (1) `pool.prepare(sql)` then `.columns()` to discover column names WITHOUT executing - an empty column list means a non-row statement (UPDATE/INSERT/DDL) -> run via `pool.execute()` and report `rows_affected()`; a non-empty list means row-returning -> (2) wrap as `SELECT <col>::text FROM (<user sql>) AS dbui_q LIMIT N` so every column decodes as text. Strip a trailing `;` before wrapping or the subquery is a syntax error. Needs `use sqlx::{Column, Executor, Statement}` for `.columns()`/`.name()`/`.prepare()`/`.execute()`.
- `row_to_json` recovers column names ONLY from rows that exist - a zero-row query emits zero keys, so the SQL result grid had no headers while the table card (which reads columns from `information_schema`) always did. Same `DataGrid`, different data. Fix: when the wrapped query returns 0 rows, run a column probe. FIRST attempt `... LEFT JOIN LATERAL (<sql>) dbui_q ON true` + `row_to_json(dbui_q)` FAILED: `row_to_json` over the join-nullable side of an unmatched LEFT JOIN is a whole-row NULL composite (emits no keys at all), not an object of null-valued keys. The working probe re-selects the lateral columns into a derived table so the composite is non-null: `SELECT row_to_json(dbui_cols)::text FROM (SELECT dbui_q.* FROM (SELECT 1) dbui_base LEFT JOIN LATERAL (<sql>) dbui_q ON true LIMIT 1) dbui_cols`. That derived row has individually-null but defined columns, so `row_to_json` emits every key (`wrap_columns_probe` + the `columns.is_empty()` branch in `run_query_postgres`). Lesson: `row_to_json(x)` where `x` is a nullable composite gives JSON `null`, not `{keys:null}` - wrap it in a derived table to force a real row.
- The above `prepare().columns()` trick is BROKEN on Postgres: `prepare()` makes Any *describe* the result, which maps every result column's type to `AnyTypeInfo` - and native types (timestamp, uuid, numeric, ...) have no Any mapping, so `prepare()` itself errors ("Any driver does not support the Postgres type PgTypeInfo(Timestamp)") BEFORE the text-cast wrap ever runs. `SELECT * FROM product` failed for exactly this reason while `fetch_table` worked (the latter never prepares - it reads column names from `information_schema` and casts to text). Fix: don't prepare arbitrary SQL on Postgres at all. Classify by leading keyword (`is_row_returning`: SELECT/WITH/VALUES/TABLE/SHOW/EXPLAIN, skipping leading `--`/`/* */` comments); execute non-row statements for `rows_affected()`; wrap row-returning ones as `SELECT row_to_json(dbui_q)::text FROM (<sql>) AS dbui_q LIMIT N` - one text column Any always decodes - then parse the JSON in Rust. Use `row_to_json` (not `to_jsonb`) to preserve column order, plus serde_json's `preserve_order` feature so the parsed `Map` keeps it. MySQL keeps the `prepare().columns()` path (its Any type mapping is fine for the describe). Adding `arbitrary_precision` to serde_json keeps large numerics/ints exact when re-stringified.
- Writing back via `sqlx::Any`: Postgres errors on a bound text value assigned to a non-text column (no implicit coercion), so an `UPDATE` must cast to the column's real type: `SET "col" = $1::<udt_name>` (read `udt_name` from `information_schema.columns`). MySQL coerces strings, so it binds plainly. Match the pk as text (`"pk"::text = $2`) to avoid per-type pk handling. This is the write-side twin of the read-side "cast everything to text" trap.
- `information_schema.columns.udt_name` (PG) gives the castable type name (`int4`, `uuid`, `timestamptz`); `data_type` is the verbose SQL-standard form. Use `udt_name` for `::cast`.
- "Filter" in a DB grid means a raw SQL WHERE to power users (they type `price > 10`), NOT substring search. We shipped substring first and it matched nothing - confirm filter semantics before building. Raw SQL can't be parameterized, so it's appended verbatim (the user owns SQL-injection in their own query against their own DB; a bad expression just returns a DB error toast).
- Any test that renders `WorkspaceLayout`/`Content`/`TableCard` now needs a `QueryClientProvider` ancestor (TableCard calls `useQuery` unconditionally, even when `enabled:false`). Shared `src/test/query-wrapper.tsx` exports `QueryWrapper`; pass it via RTL's `{ wrapper }` option (wrap the local `render` once rather than editing every call site).
- Don't hijack the global `Tab` key unconditionally. The table-card's "Tab toggles grid<->record view" listener is `window`-level + `preventDefault`; guard it to skip when `event.target` is an `<input>`/`<textarea>` so the filter field stays normally tabbable.
- The Cmd/Ctrl+K palette listener lives on `window` in `WorkspaceLayout` (`useEffect` + cleanup). It opens unconditionally (`setIsPaletteOpen(true)`, never toggles) so re-pressing while open is a no-op, and `preventDefault()` always fires for the combo to suppress the browser default. Palette commands are gated by a `when(state)` predicate per definition in `command-registry.ts` (strategy gating) - keeps the JSX free of tab-count ifology.
