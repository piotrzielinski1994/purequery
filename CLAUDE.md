## UI rules (non-negotiable)

- **Native browser context menu is suppressed app-wide** - `installBrowserDefaultGuards(window)` ([src/lib/browser-defaults.ts](src/lib/browser-defaults.ts)) mounted once in [src/app/providers.tsx](src/app/providers.tsx) preventDefaults `contextmenu` (a desktop app must never show Reload/Inspect Element). The app's own radix `ContextMenu`s are unaffected - they draw their menu + preventDefault on the trigger before this window bubble-phase guard runs, so it only swallows the native menu where no app menu handled the right-click. (Ported from `vidui`'s browser-defaults.) It ALSO preventDefaults the reserved-keyboard-shortcut slice (`isReservedBrowserShortcut`: Cmd/Ctrl + `r = + - _ 0 f g p s u`) so the WKWebView never reloads/zooms/prints/saves/view-sources or opens the native find-in-page - `f`/`g` in particular hand `Cmd+F` to the in-app find bar (see the in-app find bullet). Text-editing/OS combos (copy/paste/cut/select-all/undo/redo/quit/close) are deliberately absent so they keep working.
- **A database connects when its sidebar row is EXPANDED, not only when its card is active.** `DatabaseRow` ([src/components/workspace/tree-row.tsx](src/components/workspace/tree-row.tsx)) has a mount effect that auto-connects an expanded+idle database (fires once per mount via a ref), so a database restored expanded on launch (persisted `expandedIds`) populates its tables instead of showing a down-chevron with no children until a manual collapse+expand. The chevron toggle still connects on user-expand; the database card's `useAutoConnect` still connects the active tab. Because an expanded row that is ALSO the active tab now has two connect triggers on the same mount tick, `connect()` in [use-connection.ts](src/components/workspace/use-connection.ts) dedups via a module-level `inFlightConnects` Set (the `connectionStatus === "connecting"` guard is racy on mount - both read `idle` before setState lands). Tests reset it via `__resetInFlightConnects()` in `beforeEach` (a never-resolving connect mock would otherwise leak an id and drop the next test's connect).
- **All visual/interaction rules live in [docs/design.md](docs/design.md)** - read it before any UI change. Highlights, all enforced: NO rounded corners anywhere (`--radius` pinned to `0rem` in [src/index.css](src/index.css)); dividers are 1px and NEVER thicken/colour on hover or drag (use an invisible `::after` hit area); compact IDE density; theme tokens not hard-coded colors. The user has flagged rounded corners and thick borders repeatedly - treat either as a defect.
- **Sidebar tree drag-and-drop** (reparent databases/folders into folders, reorder siblings) mirrors the `requi` repo's `@dnd-kit/core` tree. Pure logic: [src/lib/workspace/tree-edit.ts](src/lib/workspace/tree-edit.ts) (`findNode`/`containsId`/`removeNode`/`insertNode`), [src/lib/workspace/move.ts](src/lib/workspace/move.ts) (`moveNode` + folder-parent/cycle guards), [src/lib/workspace/tree-locate.ts](src/lib/workspace/tree-locate.ts) (`projectDropPosition`/`dropTarget`/empty-zone ids); drag-state context is [src/components/workspace/tree-dnd.tsx](src/components/workspace/tree-dnd.tsx). **Only `folder` and `database` rows are draggable; `table` leaves are NOT** (ephemeral live-catalog nodes, never persisted-movable - so `moveNode` rejects a database/table parent, and `TableRow` has no drag wiring). A move flows through `WorkspaceProvider.moveNode` -> `setTree` -> the existing `onTreeChange` persist effect (no new wiring). dnd-kit pointer drags don't run in jsdom, so behavioral ACs live in the pure-lib tests; inject a crafted `TreeDndProvider` state to test cue rendering.
- **Sidebar multi-select** (Cmd/Ctrl+click toggle, Shift+click range, plain click resets) lives in `WorkspaceProvider`: `selectedIds: Set<string>` + `selectInTree(id, mode)` + `clearSelection`, with range computed over the VISIBLE rows by [src/lib/workspace/tree-select.ts](src/lib/workspace/tree-select.ts) (`flattenSelectable`/`rangeBetween`). Only folders/databases are selectable (tables excluded). Backspace/Delete (macOS delete emits Backspace) on a non-empty selection opens the shared `DeleteNodeDialog` (now takes `nodes: TreeNode[]` and renders "Delete N items?" for a multi-delete); the global window-keydown listener in [src/components/workspace/sidebar-tree.tsx](src/components/workspace/sidebar-tree.tsx) is guarded by `isEditableTarget` ([src/lib/workspace/is-editable-target.ts](src/lib/workspace/is-editable-target.ts)) so typing in an input/CodeMirror never deletes. Bulk delete flows through `removeNodes(ids)` (the single `removeNode` delegates to it). A right-click Delete on a row inside the selection deletes the whole selection.
- **Multi-select + drag** moves the WHOLE selection: dragging a row that is part of a multi-selection calls `moveNodes(ids, target)` ([src/lib/workspace/move.ts](src/lib/workspace/move.ts)); dragging an unselected row stays single (`moveNode`). `moveNodes` drops selected descendants of a selected folder (the folder carries them), inserts in tree document order, and rejects dropping into a dragged folder. It wants the RAW drop index, so `handleDragEnd` uses `rawDropTarget` (no single-node compensation) for the multi path while the single path keeps `dropTarget`'s compensation - `dropTarget` is now `rawDropTarget` + the single-node shift.
- **Sidebar/tab context menus** (ported from requi): the empty sidebar area ([src/components/workspace/sidebar-tree.tsx](src/components/workspace/sidebar-tree.tsx), `ContextMenu` wrapping the `<ul>`) -> New database / New folder at root; a folder row -> New database / New folder INSIDE it + Rename + Delete; a database row -> Connect/Disconnect + Rename + Delete (Disconnect also COLLAPSES the row - a disconnected database renders no table children, so leaving it expanded is misleading and re-expanding to reconnect would need a redundant collapse); an open content tab ([src/components/workspace/content-header.tsx](src/components/workspace/content-header.tsx)) -> Close / Close other tabs / Close all. A row's own (nested) `ContextMenuTrigger` wins over the empty-area menu, so right-clicking a row shows the row menu, not the root one. Context actions live in `WorkspaceProvider`: `addDatabase(parentId?)` / `createFolder(parentId?)` (insert inside the folder, auto-expand it; `createFolder` then opens the inline rename), `renameNode(id, name)` (renames ANY node, ignores blank), `renamingNodeId`/`beginRename`/`cancelRename`, `closeOtherTabs(keepId)`. Inline rename is the requi `RenameInput` in [src/components/workspace/tree-row.tsx](src/components/workspace/tree-row.tsx) (Enter commits, Escape cancels, blur commits, with the readyRef guard against the radix focus-teardown blur; `onPointerDown` stops the drag from starting while editing). The separate `NewFolderDialog` + `addFolder(name)` (Cmd/Ctrl+Shift+N, command palette) still exist for the root-folder path.
- **Grid row multi-select**: the shared `DataGrid` takes `selectedRows: Set<number>` + `onSelectRow(index, mode)` (NOT a single `selectedRow`) - Cmd/Ctrl+click toggles, Shift+click ranges, plain click replaces, via the pure reducer [src/lib/workspace/row-select.ts](src/lib/workspace/row-select.ts) (`nextRowSelection`). The editable table card ([table-card.tsx](src/components/workspace/table-card.tsx)) holds the selection STAMPED with the `rows` array it was made against - when `rows` change (sort/filter/paging/refetch) the stamp mismatches and the selection reads empty, so a positional bulk delete never hits the wrong rows (no reset-effect; `set-state-in-effect` is a lint error here). Bulk delete = `onDeleteRows(indices)` -> stages one delete mutation per row (reversible via Changes tab, same as single delete); shown as "Delete N rows" in the row menu and bound to Delete/Backspace (guarded by `isEditableTarget` + the grid must contain focus). The read-only SQL result grid passes a stable empty Set + no `onDeleteRows`, so it's non-selectable. RecordView focuses `selection.anchor`.
- **Table filter lives in the provider, keyed by tableId; the filter editor is remounted per table.** The APPLIED filter is `WorkspaceProvider.tableFilters: Map<tableId, string>` + `setTableFilter(tableId, filter)` (in-memory, NOT persisted) - the `TableCard` is a singleton keyed on the active node, so a content-tab switch unmounts it; keeping the filter local (as it once was) lost it on every switch. The card's editor DRAFT (`filterText`) stays local but re-seeds from the applied value via adjust-state-on-render when `tableId` changes, and the `SqlEditor` carries `key={tableId}` so its CodeMirror doc remounts fresh per table (one table's filter never lingers in another's editor). Regression tests in `table-content.test.tsx` ("filter persistence across tab switch").
- **Refresh re-fetches the open table** ([table-card.tsx](src/components/workspace/table-card.tsx)): a `RefreshCw` button in the filter bar (live tables only) + the grid-scope `refresh-table` shortcut (default `Mod+R`, `preventDefault` so it never reloads the webview, `isEditableTarget`-guarded) invalidate the `["table-rows", tableId]` / `["table-count", tableId]` / `["table-structure", tableId]` react-query keys, so LiveTable's queries refetch (a fresh SELECT appears in History, like a sort/Load-more). Keyed by tableId, so only the active table refreshes.
- **ONE data grid, always identical.** The table card and the SQL result pane MUST render rows/cells/headers with the exact same component - [src/components/workspace/data-grid.tsx](src/components/workspace/data-grid.tsx) (`DataGrid`). Never fork a second grid or diverge their styling. Read-only callers (SQL results) pass `editable={false}` + no-op edit handlers; the editable table card passes the real ones. If a grid change is needed, change `DataGrid` so both update together. (Grid visual rules are in design.md.) **MongoDB documents render through this SAME grid** - the backend flattens documents to the shared `TableRows` shape (top-level keys -> columns, `_id` first + `PK`, nested object/array -> compact JSON text, missing field -> `[NULL]`). Do NOT build a document-specific view.

- **JSON view** (third table view, alongside grid + record view): toggles an editable, foldable JSON array of the loaded rows over the grid. Pure logic is [src/lib/workspace/json-edit.ts](src/lib/workspace/json-edit.ts) - `rowsToJson`/`parseJsonRows` (ADT) + `diffToMutations` (diffs the edited array against the saved rows BY PRIMARY KEY into rowIndex-keyed intents). The view component is [src/components/workspace/json-view.tsx](src/components/workspace/json-view.tsx) (reuses the `sql-editor-theme` CodeMirror chrome + `@codemirror/lang-json` fold; read-only when given no `onSave`). The toggle is an INDEPENDENT boolean lifted to `WorkspaceProvider` (`isJsonView`/`toggleJsonView`, like `isSidebarVisible`) so the palette command and the grid-scope `toggle-json-view` shortcut (`Mod+Shift+J`) share it; record view stays local to `TableView`. **It does NOT fork the grid** (respects the "ONE data grid" rule) - it's an alternate view of the same rows. Edits AUTO-STAGE on a debounce into the EXISTING pending-edits pipeline (NO local Save/Discard - the Changes-tab pending bar is the single stage/commit gate; its Save shows the SQL before touching the DB, so the mass-delete footgun is bounded by that explicit commit, not the JSON buffer): SQL change -> `cell` per field (SQL has no whole-row `Replace` - rejected by `build_mutation`), Mongo change -> `replace`, new PK -> `insert`, missing PK -> `delete`. `saveJson` in `LiveTable` is RECONCILING + idempotent (debounce calls it on every edit): it builds each mutation's id via `jsonMutationId` (pure, in json-edit.ts - cell/delete/replace key by row identity, insert keys by PK value so a re-diff of the same new row collides), re-stages the current diff, and discards any id it previously staged that the new diff dropped (tracked in `jsonStagedIds` ref), so reverting an edit un-stages it and a re-fire never duplicates an insert. Diff rejects (stages nothing): bad JSON, SQL unknown/missing column key, duplicate PK. Draft (unsaved-insert) rows are excluded - the JSON view diffs SAVED `rows` only (`jsonRows` prop), drafts stay in the grid. **The change-diff compares CANONICAL values, not raw strings** (`canonicalCell` re-serialises the stored cell the way an edit would): a Postgres `jsonb` cell comes back as `{"vip": false, "seat": 1}` (spaces) but an edit re-emits `{"vip":false,"seat":1}` (compact) - a raw string compare flags EVERY structured row as changed (one spurious `cell` per jsonb column per row). `JsonView` is just the CodeMirror editor (NO buttons, NO persistent status bar - an error bar appears below it ONLY on bad-JSON/validation failure); it fills height directly (own scroll) and is NOT wrapped in the grid's `ScrollArea` (that grew the editor to full content height) - the `ScrollArea` lives inside `TableView` around the grid/record branches only.

- **Read-only connection mode** (F11 - prod safety): a per-database `readOnly: boolean` flag ON the database node (default false), edited via the `ReadOnlyField` square switch in the Settings tab (under accent color) and persisted in `workspace.json` (mirrors `accentColor` - `mergeReadOnly` keeps only boolean `true`, `hydrate` defaults missing to false, `dehydrate` OMITS false). Provider action `setDatabaseReadOnly(id, ro)` mirrors `setDatabaseAccent` exactly. When ON it blocks EVERY frontend write path at the boundary - **no backend change**: (1) table mutations fold into the ONE existing `editable` gate in [table-card.tsx](src/components/workspace/table-card.tsx) - `editable = primaryKey !== null && !readOnly` - so inline edit / Add-row / delete / bulk-delete / clone / Mongo doc-edit / JSON-view auto-stage all disable together via the SAME no-PK read-only rendering (no new disabled state), and the table Save bar is hidden (`save()` also early-returns); (2) a write-shaped command in [sql-tab.tsx](src/components/workspace/sql-tab.tsx) `submit` is rejected before `run.mutate` with a sticky warning toast + a History error line, using the pure guard picked by engine ([dispatch.ts](src/lib/script/dispatch.ts)): `isWriteSql` (leading SQL keyword, the SAME guard the Script tab uses) for the SQL engines, `isWriteMongo` (the `.op(` after `db.<coll>.` matched against `MONGO_WRITE_OPS`) for MongoDB - because the Mongo Query tab now runs writes too (updateOne/insertOne/...), so it is NO longer read-only-by-default at the backend. There is NO other frontend DB-send path (Changes tab has Discard only, no global Save). Prefix-only like the Script tab: a read-led multi-statement buffer chaining a write is a known, documented gap. When adding a NEW write path, gate it on the owning node's `readOnly` the same way.

- **Default schema** (per-database sidebar filter): a per-database `defaultSchema: string | null` flag ON the database node (default null), edited via the `DefaultSchemaField` Select in the Settings tab (under Manual commit) and persisted in `workspace.json` - mirrors `readOnly` at every site EXCEPT it holds a string not a bool (`mergeDefaultSchema` keeps only a NON-EMPTY string, `hydrate` defaults null, `dehydrate` OMITS null/empty; provider `setDatabaseDefaultSchema(id, schema|null)` + `setDefaultSchema` tree map mirror `setDatabaseReadOnly`/`setReadOnly`). **No backend change; FK-nav/quick-open untouched.** Two effects: (1) TREE DISPLAY - `DatabaseRow` ([tree-row.tsx](src/components/workspace/tree-row.tsx)) filters `node.tables` through the pure `visibleTables(tables, defaultSchema)` ([tree-schema.ts](src/lib/workspace/tree-schema.ts)) - a **STRICT** filter (only tables whose `.schema` equals it; a stale/absent schema shows ZERO rows, by design) - and forces the BARE table label (`multiSchema = defaultSchema === null && isMultiSchema(tables)`, so the `schema.table` prefix never shows once pinned to one schema). (2) SQL AUTOCOMPLETE - the pin flows as the `SqlEditor` `defaultSchema` prop (`SqlPane` -> `SqlEditor`), threaded into `buildNamespace(schema, pinnedSchema, engine)` so lang-sql's `schemaCompletionSource` reports it as the default schema: its tables complete UNQUALIFIED (no `schema.` prefix). Falls back to the engine's BUILTIN default when unset or not in the live catalog - `public` for Postgres, `dbo` for SQL Server (`BUILTIN_DEFAULT_SCHEMA` in sql-editor.tsx; other engines have no schema level). The single-line filter row is unaffected (it scopes to `defaultTable`). The selector options are `schemaOptions(node.tables)` (distinct non-null schemas, sorted) + an "All schemas" (null) choice via the `ALL_SCHEMAS = "__all_schemas__"` sentinel (radix SelectItem forbids an empty-string value); the trigger renders `{defaultSchema ?? "All schemas"}` VERBATIM so a stale/disconnected saved schema stays visible + selectable (appended to the option list when not in the live set). Postgres + SQL Server in effect (MySQL/SQLite/Mongo tables carry `schema: null`). **`applyDatabaseConfig` in [workspace-context.tsx](src/components/workspace/workspace-context.tsx) destructures+rebuilds a node on connect, so it MUST carry `defaultSchema` through** (else a connect wipes it - same hazard for any new node field). Every full `DatabaseNode` test-literal needs `defaultSchema: null` (same fixture rule as readOnly/manualCommit).

- **Manual-commit transaction mode** (F12 - prod safety): a per-database `manualCommit: boolean` flag ON the database node (default false), edited via the `ManualCommitField` square switch in the Settings tab (UNDER Read-only) and persisted in `workspace.json` - a byte-for-byte mirror of F11 `readOnly` (`mergeManualCommit` keeps only boolean `true`, `hydrate` defaults false, `dehydrate` OMITS false; provider `setDatabaseManualCommit(id, bool)` + `setManualCommit` tree map mirror `setReadOnly`). This is REAL auto-commit-OFF (DBeaver/DataGrip-style), NOT a confirm-before-send dialog (that earlier F12 reading was reverted - see adr.md 2026-07-11). **SQL engines only** - MongoDB is rejected (its NoSQL driver, like DBeaver's, has no manual-commit toolbar; Mongo tx needs a replica set). **Backend is the core** (first stateful-across-commands element, [src-tauri/src/db.rs](src-tauri/src/db.rs)): a `TRANSACTIONS` registry (`HashMap<id, Arc<tokio::Mutex<TxSession>>>`) pins a `PoolConnection` from the held pool. `begin_transaction(id)` (idempotent) acquires it + issues literal `BEGIN`; while a tx is open, the `acquire_conn(id)` seam returns the PINNED connection (a `ConnHandle` deref'ing to `&mut AnyConnection`) for EVERY connection-addressed command instead of a fresh pool one - so reads see uncommitted writes. This forced the read/mutation helpers (`run_query_batch`/`read_table_rows`/`count_table_rows`/`fetch_schema`/`read_table_structure`/`apply_mutations` + `fetch_introspection`) to take `&mut AnyConnection` not `&AnyPool`. `commit_transaction`/`rollback_transaction` run `COMMIT`/`ROLLBACK` + drop the connection back to the pool; `disconnect_database` auto-`ROLLBACK`s an open tx (never leaks the pinned conn; pool max is 5). `transaction_state(id)` (sync, false for unknown/Mongo) drives the UI. New Tauri commands in [lib.rs](src-tauri/src/lib.rs) (Mongo path returns Err). **NO statement auto-wrap** - the tx is an explicit session the user opens by writing (mirrors the "multi-statement runs are user-owned transactions" ADR). Frontend: the write paths call `beginTransaction(id)` BEFORE sending when the node is `manualCommit` - **`await` it in [sql-tab.tsx](src/components/workspace/sql-tab.tsx) `submit`** (an un-awaited begin races `execute_sql` onto a fresh pool conn -> write escapes the tx; regression test `manual-commit-sql.test.tsx`); table Save in [table-card.tsx](src/components/workspace/table-card.tsx) awaits it too. A content-header **Commit/Rollback** toolbar + `* uncommitted changes` cue ([content-header.tsx](src/components/workspace/content-header.tsx), `ManualCommitControls`) renders only when the active db is `manualCommit` AND the `["tx-state", dbId]` react-query is true. **Commit opens a confirm modal listing the exact statements the transaction will persist** (Rollback fires directly - destructive but instantly reversible by re-running); the modal renders them through `SqlText` (see the SQL-highlight note below). The statement list is `WorkspaceProvider.txStatements: Map<dbId, string[]>` (in-memory, mirrors `tableFilters`) - each write APPENDS its SQL via `appendTxStatement` **only after it SUCCEEDS** (SQL Run appends in the mutation `onSuccess`, filtered by `isWriteSql`, keyed by `outcome.statement`; table Save appends each `savedSqls` line after `applyRowMutations` resolves ok), so a failed/retried attempt is NEVER listed. Cleared by `clearTxStatements` on commit/rollback/disconnect. NOTE the statements have ALREADY run inside the open tx - the modal shows what COMMIT will make permanent, it does not re-run them. Commit/Rollback call the commands, `setQueryData(["tx-state"], false)` authoritatively (not a refetch race), and invalidate `["table-rows"]`/`["table-count"]` so a rollback's discarded rows leave the grid. readOnly composes inertly (a read-only DB never writes, so no tx begins). **A failed statement inside an open tx does NOT poison it**: inside the pinned tx each statement/mutation is wrapped in a `SAVEPOINT purequery_stmt` (`run_one_in_savepoint` / `execute_mutation_in_savepoint`, gated on `ConnHandle::is_pinned`) - a failure `ROLLBACK TO SAVEPOINT` undoes just that statement so the next command still works, mirroring DBeaver's per-statement savepoint (without it Postgres marks the whole tx aborted -> "current transaction is aborted, commands ignored" on every later command). The pooled/auto-commit path takes no savepoint (a savepoint outside a tx is an error). Only disconnect force-rolls-back the whole tx. SQLite does NOT abort a tx on a statement error (that symptom is Postgres-specific) so the SQLite `tx_tests` are a savepoint SMOKE; the real recovery + cross-connection MVCC isolation are `#[ignore] live_pg_*` cargo tests (docker stack, port 55432).

- **Structure view** (F6 #14 - fourth table view): a READ-ONLY metadata panel over the grid showing the open table's Columns (name/type/nullable/PK/default/ordinal), Indexes (name/fields/unique/primary), Foreign keys, and Constraints (check+unique). Component is [src/components/workspace/structure-view.tsx](src/components/workspace/structure-view.tsx); it is NOT the shared `DataGrid` (metadata, not editable rows - so the "ONE data grid" rule doesn't apply, it's an intentionally separate `<table>`). Data is fetched via `fetchTableStructure` react-query (`staleTime: Infinity`, keyed `["table-structure", tableId]`); only LIVE (connected) tables have the view - the static/mock path never reads `isStructureView`. **The `enabled` gate is `isStructureView || !isMongo`**: a SQL table fetches its structure EAGERLY (not gated on the view) because FK navigation (F13) needs the foreign keys before the Structure view is ever opened - the FK header marker + "Go to" row menu draw on first render; Mongo stays lazy (no FKs, nothing needs it early). `StructureBranch` renders loading/error/data; each empty section shows "None". - **FK navigation** (F13 - SQL only): a live SQL table row's right-click menu shows a "Go to `<refTable>` (`col=value`)" item per outbound foreign key whose local column value(s) are all non-null; selecting it `openNode`s the referenced table's tab + `setTableFilter`s a WHERE fragment pinning the referenced row (`<refCol> = <value>` AND-joined). FK local columns are marked `FK` in the grid header (alongside `PK`/`NN`), and an FK cell's value renders as a **Cmd/Ctrl+click link** (dotted underline, `--primary`) that fires the SAME navigation - `foreignKeyForColumn`/`isForeignKeyNavigable` ([foreign-key-nav.ts](src/lib/workspace/foreign-key-nav.ts)) pick the fk per cell; the modified click `stopPropagation`s so it never also toggles the row selection, a plain click selects the row like any cell (visual rule in design.md). Pure logic: [src/lib/workspace/foreign-key-nav.ts](src/lib/workspace/foreign-key-nav.ts) (`navigableForeignKeys` = non-null-value FKs + label; `fkTargetTableId` = the `${databaseId}::${referencedSchema ?? ""}::${referencedTable}` node-id formula) + `fkFilter` in [query-preview.ts](src/components/workspace/query-preview.ts) (engine-quoted idents + `sqlLiteral`-escaped values). The `DataGrid` takes `foreignKeys` + `onFollowForeignKey`; `LiveTable` owns `followForeignKey` (resolves the target id, `toast.error` if not in `nodesById`, else `navigateTo` - NO `readOnly` gate, navigation is a read). A row can carry MULTIPLE FKs to DIFFERENT tables (one "Go to" item each); a COMPOSITE FK (>1 column) is ONE item joining every `refCol=value` pair. **The PG FK introspection query correlates the referenced column by `position_in_unique_constraint` against `key_column_usage` (NOT `constraint_column_usage`)** - the latter joins only on constraint_name and fans a composite FK into a cartesian product (2-col FK -> 4 rows -> duplicated pairs in the label); `db.rs`'s `foreign_key_query` test pins this. **Requires `referencedSchema` on `ForeignKey`** (Rust `db.rs` + TS `model.ts`): the PG FK query selects `ccu.table_schema`, MySQL/SQLite select `NULL`, so a cross-schema Postgres FK resolves to the correct node. Mongo has no FKs so no items/marker appear.
  - **Back/forward navigation**: FK jumps go through `WorkspaceProvider.navigateTo({tableId, filter})`, which pushes onto a `(tableId, filter)` history stack (pure reducer [src/lib/workspace/nav-history.ts](src/lib/workspace/nav-history.ts) - `pushNavigation` seeds the source as the base entry so Back returns to it, drops forward history on a new jump, no-ops a re-jump to the current position). `goBack`/`goForward` walk the cursor, restoring BOTH the active tab (`openNode`) and that table's filter (`setTableFilter`). Exposed as `canGoBack`/`canGoForward` + the `nav-back`/`nav-forward` shortcuts (`Mod+[` / `Mod+]`, global scope) + `Navigate back`/`Navigate forward` palette commands (Tabs group, gated on `canGoBack`/`canGoForward`). There are NO content-header arrow buttons - the user removed them; navigation is palette + keymap only. In-memory only. (This replaced the original "rely on the tab bar" decision - see adr.md.)

- **Mock data generator** (F17): a "Generate mock data" palette command (Create group, gated on `isTableActive`) opens the `MockDataDialog` ([src/components/workspace/mock-data-dialog.tsx](src/components/workspace/mock-data-dialog.tsx)) over the open live table. Pure logic is [src/lib/workspace/mock-data.ts](src/lib/workspace/mock-data.ts): `autoStrategy(column)` (type/name/PK -> a default `MockStrategyKind`) + `generateRows(configs, count, seed)` (a **seeded deterministic** mulberry32 PRNG -> ADT `Result`; validates count against `MAX_MOCK_ROWS=200` + rejects an empty enum list; a `skip` column is OMITTED from each row object, a `null` column present-as-null - the two are DISTINCT). ## Engines: SQL (sqlx) vs MongoDB vs SQL Server vs DynamoDB (four-family dispatch seam)

- The sqlx SQL engines (Postgres/MySQL/SQLite) run on `sqlx::Any` in [src-tauri/src/db.rs](src-tauri/src/db.rs). **MongoDB is NOT a `DbEngine`/`sqlx` arm** - it lives in [src-tauri/src/mongo.rs](src-tauri/src/mongo.rs) (the `mongodb` crate) with its own client registry. `sqlx::Any` cannot drive Mongo; never add a `DbEngine::Mongo` to `db.rs`.
- **SQL Server is also NOT a `db.rs`/`sqlx::Any` arm** (F-MSSQL) - sqlx dropped its MSSQL driver in 0.7, so SQL Server lives in [src-tauri/src/mssql.rs](src-tauri/src/mssql.rs) on the pure-Rust **`tiberius`** TDS crate with its OWN registry, mirroring `mongo.rs`. It IS relational SQL, so it produces the SAME shared IPC structs (`TableRef`/`TableRows`/`QueryOutcome`/`RowMutation`/`TableStructure`/`DatabaseObject`) and the whole frontend composes with no forked UI - the bulk is the backend adapter + per-engine T-SQL builders (`browse_query`/`count_query`/`quote_ident` (`[..]`)/`qualified_name`/mutation builders/the four structure builders + folds/`database_objects_query`/`cell_from_column`). **tiberius's `Client` is a SINGLE connection** (methods take `&mut self`), not a pool: the registry is `HashMap<id, Arc<tokio::Mutex<MssqlConn>>>` and every command locks it, so concurrent commands on one id serialise. This makes manual-commit tx SIMPLER than the sqlx `TxSession` - the held client is already the pinned connection, so a tx is just `BEGIN TRANSACTION` + a `tx_open` flag (per-statement recovery via `SAVE TRANSACTION purequery_stmt` - but SQL Server's default `XACT_ABORT OFF` does NOT abort the whole tx on an ordinary statement error, verified: like SQLite, unlike Postgres, so the savepoint is DEFENSIVE for `XACT_ABORT ON`/fatal errors + DBeaver parity, and the `live_mssql` tx-recovery step is a SMOKE not a proven-red recovery); `transaction_state` reads a PARALLEL synchronous `TX_OPEN` set (NOT the per-id tokio mutex, which a long in-tx statement holds - a `try_lock` there would wrongly report "no tx"). `cell_from_column` maps each tiberius `ColumnData` variant to display text (temporal types via chrono `FromSql`; an unknown variant falls back to a `{:?}` debug string, never panics). Backup adds a `BackupSpec::Mssql` arm reusing `mssql::read_table_page`/`list_tables`/`open_standalone` (its `mssql_sql_literal` doubles `'` but NOT `\` - SQL Server is not MySQL). To add a NEW mssql feature, add the pure T-SQL builder + a `lib.rs` `mssql::is_connected` dispatch arm the same way. Oracle (native Instant Client libs) + Redis (breaks the one-grid model) are deferred, NOT built.
- **DynamoDB is a FOURTH family** (F-DYNAMO) - a key-value/NoSQL store with no sqlx driver, so it lives in [src-tauri/src/dynamo.rs](src-tauri/src/dynamo.rs) on the AWS SDK (`aws-sdk-dynamodb`/`aws-config`/`aws-credential-types`) with its OWN registry, mirroring `mongo.rs`. The registry holds the SDK `Client` DIRECTLY (`HashMap<id, Client>`, clone-out - the client is `Arc` internally and DynamoDB is a stateless HTTP service, so NOT the mssql `Arc<Mutex>` single-conn pattern; there is no held socket or target-db name to carry). The **Query tab runs PartiQL through the EXISTING `execute_sql` command** (a held-dynamo id routes `execute_sql` -> `dynamo::run_query`) - NO new Tauri command, NO FE executor branch (the SQL tab already calls `executeSql` for every non-mongo engine), and `isWriteSql` already covers PartiQL `INSERT/UPDATE/DELETE` so the read-only gate composes. It emits the SAME shared IPC structs so the grid/editor/structure UI is unforked; the FE cost is the `DbEngine` union + a NEW `DynamoConnection`/`PersistedDynamoDatabase` shape (DynamoDB is NOT a `NetworkEngine` - it carries `region`/`accessKeyId`/`secretAccessKey`/optional `sessionToken`/`endpoint`, not host/port/user/password; blank keys -> the default AWS credential chain, `endpoint` overrides for dynamodb-local). **Engine-driven divergences (all documented):** the "database" node = one AWS region (`schema:null`, no views, no db/schema level); header sort is DISABLED (`onSortColumn={undefined}` for a dynamo table - no general ORDER BY); the status-bar count is the APPROX `DescribeTable.ItemCount` rendered `~N` (NOT a scan-count); paging is the opaque `LastEvaluatedKey`/PartiQL `nextToken`, threaded via a nullable `TableRows.nextToken` (skip-serialized, so the offset-paged SQL/Mongo engines are unaffected) + a `fetch_table` `nextToken` param, and `useInfiniteQuery` branches on `isDynamo` (pageParam = the token, not an offset). **v1 CRUD is SIMPLE-KEY inline only:** a composite (partition+sort) table renders READ-ONLY (`flatten_items` returns `primary_key: None` for a composite key -> the shared PK-less gate; the mutation builders `reject_composite`) but stays writable via PartiQL; full composite inline-CRUD needs extending the shared single-`pk_value` `RowMutation` and is deferred. Manual-commit/FK-nav/object-tabs/default-schema are rejected/absent like the Mongo gaps (`lib.rs` returns `Err`/`false`/`vec![]`; the Settings form hides Manual-commit + Default-schema for dynamo). Backup is a `.jsonl` item-per-line export (`BackupSpec::Dynamo` + `run_dynamo_backup` + `estimate_dynamo_rows` = the `DescribeTable.ItemCount` sum guardrail). **`AttributeValue<->serde_json::Value` is HAND-ROLLED** (`attribute_to_value`/`attribute_to_cell`) - the crate-wide `arbitrary_precision` serde_json feature breaks a blanket bridge, the SAME reason `mongo.rs` hand-rolls `Value->Bson`. PartiQL highlights via the generic `StandardSQL` `@codemirror/lang-sql` dialect; icon is `DiAws` (react-icons/di - there is NO DynamoDB-specific glyph, `SiAmazondynamodb` is absent). Live smoke `live_dynamo_*` runs against `amazon/dynamodb-local` (test-stack host port 8009). To add a NEW dynamo feature, add the pure builder/mapper + a `lib.rs` `dynamo::is_connected` dispatch arm the same way.
- [src-tauri/src/lib.rs](src-tauri/src/lib.rs) is the dispatcher: every connection-addressed command routes to the Mongo path when `mongo::is_connected(id)`, the **mssql path when `mssql::is_connected(id)`**, the **dynamo path when `dynamo::is_connected(id)`**, else the sqlx SQL path. `connect_database` takes a raw `serde_json::Value` and peeks the `engine` tag (`mongodb` -> `MongoConfig`, `sqlserver` -> `MssqlConfig`, `dynamodb` -> `DynamoConfig`, else the SQL `ConnectionConfig` enum). Keep new per-connection commands dispatching the same FOUR ways (DynamoDB PartiQL rides `execute_sql`, so that command's dynamo arm precedes the mssql/SQL arms). FE-side, `sqlserver` is a `NetworkEngine` (same host/port/user/password shape as Postgres/MySQL, default port 1433) - add it to every exhaustive `Record<DbEngine, ...>` map (`tsc` catches misses) + the `NETWORK_ENGINES` set (persistence hydration) + the `sql-editor` `dialects` map (`MSSQL` from `@codemirror/lang-sql`). Icon is `DiMsqlServer` (react-icons/di - simple-icons has no SQL Server glyph). **`dynamodb` is NOT a `NetworkEngine`** - it needs its OWN `DynamoConnection` type + `PersistedDynamoDatabase` shape + `mergeDatabaseFile`/`hydrateDatabase`/`dehydrateDatabase` branch (like Mongo/SQLite, which also aren't network); add it to every exhaustive `Record<DbEngine, ...>` map (`tsc` catches ENGINE_LABELS/ENGINE_ICONS/KINDS_BY_ENGINE/backup ext+label) AND the silent-miss non-exhaustive maps (the `as const` `dialects` -> `StandardSQL`; `DEFAULT_PORT_BY_ENGINE`/`BUILTIN_DEFAULT_SCHEMA` are `Partial` - leave dynamo absent; the hardcoded settings-tab `<SelectItem>` list).
- The Mongo path returns the SAME IPC structs as SQL (`TableRef`/`TableRows`/`QueryOutcome`/`RowMutation`/`TableStructure`) so the frontend stays engine-agnostic. The shared `RowMutation::Replace` (full-document `replaceOne`) is Mongo-only - the SQL `build_mutation` returns `Err` for it.
- **Schema introspection (F6)**: `connect_database` returns `ConnectCatalog { tables, views }` (NOT a bare `Vec<TableRef>`) - `views_query` (`table_type='VIEW'` PG/MySQL, `type='view'` SQLite) runs alongside `catalog_query` on connect; a views-query failure degrades to empty rather than failing the connect; Mongo returns no views. `fetch_table_structure(id, schema, table)` -> `TableStructure` (columns/indexes/foreign_keys/constraints), dispatched Mongo-vs-SQL in `lib.rs` like every command; SQL builds it from four pure per-engine query builders (`structure_columns_query`/`index_query`/`foreign_key_query`/`constraint_query`, each `(engine, has_schema)`) + the `fold_indexes`/`fold_foreign_keys` pure grouping helpers (composite index/FK = one-row-per-column folded by name); Mongo fills indexes only via `list_indexes`. SQLite check-constraints are NOT exposed by pragma (they live in DDL text) so `constraint_query` returns UNIQUE only there.
- **File logging is emitted at the `lib.rs` dispatcher, not inside `db.rs`/`mongo.rs`** - one call site per command (connect/disconnect/execute_sql/execute_mongo/apply_mutations) covers BOTH engines with no duplicate lines. Each site wraps the awaited call in `std::time::Instant`, then `log::info!`/`error!` a `key=value` line built by a PURE formatter in [src-tauri/src/logging.rs](src-tauri/src/logging.rs) (`format_connect_ok`/`_err`, `format_disconnect`, `format_query_ok`/`_err`, `format_mutations`) - the formatters take primitives + pre-computed millis (clock-free, unit-tested), mirroring the `launch_log_name` pure/impure split. A cancelled connect/query (the `db::CANCEL_SENTINEL` string) logs NO error line - guard with `is_cancel_sentinel` before the `log::error!` (same rule the FE uses to suppress its toast). One line PER command invocation, not per `;`-statement (the History tab already gives per-statement granularity in-app). Logs land in the per-launch `purequery-<ts>.log` (`logging::init`, best-effort) + stdout. To log a new command, add a formatter + a dispatcher call site the same way; don't sprinkle `log::` through the engine modules.
- Build BSON from JSON via the hand-rolled `Value -> Bson` map in `mongo.rs`, NOT `bson::to_document`: the crate-wide serde_json `arbitrary_precision` feature makes `to_document` emit a `$serde_json::private::Number` wrapper Mongo rejects.
- **Read-only SQL shown in a plain-text list is syntax-highlighted via `SqlText`** ([sql-text.tsx](src/components/workspace/sql-text.tsx)), NOT CodeMirror: the History log, the Changes list, and the manual-commit Commit modal all render their SQL through it. `SqlText` maps the pure `highlightSql` tokenizer ([src/lib/workspace/sql-highlight.ts](src/lib/workspace/sql-highlight.ts) - a coarse keyword/string/number/comment/identifier splitter, engine-agnostic, never throws) to `<span>`s coloured with the SQL editor's theme `editorColors`, so it reads like the editor WITHOUT mounting a CodeMirror per row (a list of dozens of statements can't each be an editor). The live editors (SQL tab, JSON view) keep real CodeMirror; the deleted `SqlPreview` (a CM-in-the-modal experiment) was replaced by `SqlText` so there is ONE display highlighter. Gotcha for tests: `SqlText` splits a statement into per-token spans, so `getByText(/whole SQL/)` fails ("broken up by multiple elements") - assert on a row's `textContent` (see `sqlText()` matcher in `console.test.tsx`, or `.getAllByRole("listitem").filter(n => /.../.test(n.textContent))`).
- Frontend per-engine differences (History preview strings, filter syntax) live behind `queryPreview(engine, schema, resolveType?)` in [src/components/workspace/query-preview.ts](src/components/workspace/query-preview.ts) + a `mongodb` JSON CodeMirror mode in [src/components/workspace/sql-editor.tsx](src/components/workspace/sql-editor.tsx). Add to the strategy, don't sprinkle `engine === "mongodb"` through the table card. **The SQL preview strings are TYPE-AWARE**: `queryPreview`'s optional `resolveType(column) -> dataType` lets `sqlValue` emit a numeric/boolean column's value as a BARE literal (`123`, `true`, `NULL`) instead of a quoted string, so the read-only preview editor colours it as a number/keyword (not string-green) and reads like the typed bind actually sent. `TableCard` supplies the resolver from the fetched column metadata via a `columnTypesRef` (the ref breaks the memo/columns init-order + the resolver reads it lazily at edit/save/copy time, never during render - hence the one `react-hooks/refs` disable). Omitting `resolveType` (Copy-as-SQL, filter-only, Mongo, FK `fkFilter`) keeps the always-quote `sqlLiteral` behaviour. A numeric column whose value is non-numeric falls back to a quoted literal (never emits invalid SQL). This is PREVIEW-only cosmetics; the real write is a typed parameterized bind in `db.rs` (`$n::type` / `?`) and was always correct regardless of preview quoting.
- The Mongo **Query tab reuses the SQL editor pane** (`SqlTab`/`SqlPane`): same saved-script document tabs, Run/Cancel, History. `SqlTab` picks the executor by `node.engine` (`executeMongo` vs `executeSql`). Mongo commands are self-contained `db.<coll>.<op>(...)` (collection in the text, no picker), `;`-separated; backend `parse_command` + `run_command`/`run_query` in `mongo.rs`. **Both reads AND writes run**: reads `find`/`aggregate` (capped by `limit`, return rows); writes `insertOne`/`insertMany`/`updateOne`/`updateMany`/`deleteOne`/`deleteMany`/`replaceOne` (return an affected count, `returns_rows:false`). `parse_command` splits the args between the outer parens on TOP-LEVEL commas (`split_args`, brace/bracket/string-aware) so two-arg writes (`updateOne(filter, update)`) parse; `MongoCommand.args: Vec<String>` (was a single `arg`). Each write arg is parsed to a BSON document via the SAME `json_object_to_document`/`validate_extended_json`/Extended-JSON path the filter uses (so `{"$oid":...}` works in a filter). Writes are blocked upstream when the DB is read-only (see F11 / `isWriteMongo`), never in `run_command`. Do NOT reintroduce a collection picker or a separate mongo query component.
- Mongo **autocomplete**: in the Query tab `db.` -> collection names, `db.<coll>.` -> the read + write ops (find/aggregate/insertOne/insertMany/updateOne/updateMany/deleteOne/deleteMany/replaceOne), inside a read OR write command body after a `"` -> that collection's sampled field names; in the **filter row** (a bare find document, no `db.<coll>.<op>` prefix) after a key-opening `"` -> the fields of the collection the card is scoped to (`defaultTable`). Fields come from `mongo::fetch_schema` (samples ~50 docs/collection on connect -> `TableSchema` per collection) flowing through the same `databaseSchemas` -> `schema` prop as SQL. Completion lives in `mongoCommandSource` in [src/components/workspace/sql-editor.tsx](src/components/workspace/sql-editor.tsx) - it resolves the collection from the `db.<coll>.<op>` prefix, else falls back to `defaultTable`.
- ObjectId/date `_id` (or any BSON type) in a filter uses **Extended JSON**: `{"_id":{"$oid":"..."}}`, `{"$numberLong":"42"}`, `{"$date":"..."}`. Plain JSON can't match an ObjectId. Decoded in `json_value_to_bson`/`extended_json_type`; a malformed wrapper is a filter error.
- Live Mongo smoke (needs the docker test-stack up): `cargo test --manifest-path src-tauri/Cargo.toml live_mongo -- --ignored`. Test-stack Mongo is host port 27018 (root user `purequery`/`purequery`, `authSource=admin`).

## Features

- Each feature lives in its own folder: `docs/features/<timestamp>-<slug>/`.
  - `<timestamp>` = `YYYYMMDDHHMMSS` (creation time). `<slug>` = short kebab-case name.
  - Example: `docs/features/20260618193518-bootstrap/`.
- Every feature folder holds two files:
  - `spec.md` - what + why. Follows the spec template structure (overview, acceptance criteria, user test cases, data model, edge cases, dependencies).
  - `plan.md` - how. Follows the plan template structure (task breakdown, execution order, file changes, acceptance verification).
- Adding a new feature:
  1. Create the folder with current timestamp + slug.
  2. Write `spec.md` first. Get it approved before planning.
  3. Write `plan.md` from the approved spec.
  4. Log any significant choices made while specing to [docs/adr.md](docs/adr.md).
- Branch naming: when working on a feature (not a quick fix), the branch name must match the feature's folder name under `docs/features/` exactly (e.g. folder `20260618223203-layout` -> branch `20260618223203-layout`). Quick fixes are exempt.

## Architectural Decisions

- Log only significant, costly-to-reverse or contested decisions to [docs/adr.md](docs/adr.md).
- Significant = changes architecture/data model, hard to undo later, or had real alternatives debated. NOT routine config (script aliases, package manager, default lib options).

## Before committing

- Check whether the change makes README.md or CLAUDE.md drift:
  - New script / removed dependency / renamed module -> update README.
  - New convention or gotcha that future-you would miss -> add to CLAUDE.md (or docs/learnings.md).
  - Removed feature or file referenced in either doc -> remove the reference.
- No duplicates between README.md and CLAUDE.md. Each fact lives in exactly one place:
  - README.md = onboarding facts a human needs to run the app: install steps, commands, repo layout sketch.
  - CLAUDE.md = working rules for an agent editing this repo: conventions, gotchas, "how to add a feature", invariants.
  - If a fact would fit both, put it in CLAUDE.md and link from README only if a human reader needs the pointer.
- If neither doc needs to change, say so explicitly in the pre-commit summary so it's a deliberate decision, not an oversight.

## TDD

Write code red-green-refactor:

1. Red - add a failing test that pins the behaviour you want. Run the relevant suite and confirm it fails for the right reason (not a typo, not a missing import).
2. Green - write the smallest production change that makes it pass. No speculative branches, no helper extraction yet.
3. Refactor - once green, clean up names, extract duplication, tighten types. Tests stay green throughout.

Two test layers, pick the one that owns the behaviour:
- Frontend (React/TS) -> `npm test` (Vitest).
- Rust backend / Tauri commands -> `cargo test` in `src-tauri/`.

Don't skip red. A test that's never seen failing is a test you can't trust. Don't refactor on red - get to green first, then improve.
